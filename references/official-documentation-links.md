### 3.4 Link Validation Script

```python
#!/usr/bin/env python3
"""
link_checker.py — Validate all URLs in official-documentation-links.md
Usage: python link_checker.py references/official-documentation-links.md
"""
import re
import requests
import sys
from concurrent.futures import ThreadPoolExecutor, as_completed
from urllib.parse import urlparse

def extract_urls(filepath):
    """Extract all URLs from a markdown file."""
    with open(filepath, 'r', encoding='utf-8') as f:
        content = f.read()
    # Match URLs in markdown table rows and inline links
    urls = set(re.findall(r'https?://[^\s|)>]+', content))
    # Strip trailing punctuation
    urls = {u.rstrip('.,;') for u in urls}
    return sorted(urls)

def check_url(url):
    """Check if a URL is reachable. Returns (url, status_code, ok)."""
    try:
        response = requests.head(url, allow_redirects=True, timeout=10,
                                  headers={'User-Agent': 'Mozilla/5.0'})
        # Some docs servers don't support HEAD; try GET
        if response.status_code == 405:
            response = requests.get(url, allow_redirects=True, timeout=10,
                                     headers={'User-Agent': 'Mozilla/5.0'},
                                     stream=True)
            response.close()
        return (url, response.status_code, response.status_code < 400)
    except requests.RequestException as e:
        return (url, str(e), False)

def main():
    filepath = sys.argv[1]
    urls = extract_urls(filepath)
    print(f"Checking {len(urls)} URLs...\n")

    broken = []
    with ThreadPoolExecutor(max_workers=10) as executor:
        futures = {executor.submit(check_url, url): url for url in urls}
        for future in as_completed(futures):
            url, status, ok = future.result()
            status_icon = "✅" if ok else "❌"
            print(f"  {status_icon} [{status}] {url}")
            if not ok:
                broken.append((url, status))

    print(f"\n{'='*60}")
    print(f"Total: {len(urls)}, OK: {len(urls) - len(broken)}, Broken: {len(broken)}")

    if broken:
        print("\nBroken URLs:")
        for url, status in broken:
            print(f"  ❌ {url}  (status: {status})")
        sys.exit(1)

if __name__ == "__main__":
    main()
```

## 4. Feature Evaluation Process

When a new feature is identified (from release notes, blog posts, or documentation updates), it follows this evaluation pipeline before adoption.

### 4.1 Process Flow

```
MONITOR → ASSESS → SPIKE → SYSTEM TEST → DOCUMENT → REVIEW → ADOPT
```

### 4.2 Stage Details

#### Stage 1: Monitor

- **Input**: Monthly review, blog subscriptions, release notes
- **Action**: Log the feature in the **Feature Evaluation Tracker** (Azure DevOps board or spreadsheet)
- **Output**: Feature registered with source link and initial description

#### Stage 2: Assess

- **Input**: Feature description from Stage 1
- **Action**: Answer these questions:
  - Does this feature solve an existing problem we have?
  - Is this feature available in our current runtime/version?
  - What is the effort to evaluate (hours/days)?
  - What is the risk of not adopting?
- **Output**: Go/No-Go decision for spike. If Go, assign a spike owner.

#### Stage 3: Spike

- **Input**: Go decision from Stage 2
- **Action**: Build a minimal proof-of-concept in a dev workspace
  - Time-boxed: 1–2 days maximum
  - Test against a representative dataset (not production)
  - Document findings: what works, what doesn't, surprises
- **Output**: Spike report (1-page document) with recommendation

#### Stage 4: System Test

- **Input**: Positive spike report from Stage 3
- **Action**: Integrate the feature into a test pipeline
  - Run full integration tests
  - Run performance benchmarks (compare before/after)
  - Test failure scenarios (what happens when the feature fails?)
- **Output**: Test results with performance comparison

#### Stage 5: Document

- **Input**: Successful system test from Stage 4
- **Action**: Create or update documentation:
  - Update relevant reference documents
  - Create an Architecture Decision Record (ADR) if the feature changes the architecture
  - Update runbooks if operational procedures change
  - Update this link registry if new documentation is relevant
- **Output**: Documentation PR reviewed and merged

#### Stage 6: Review

- **Input**: Documentation from Stage 5
- **Action**: Present to the team for final review
  - Demo the feature in the test environment
  - Discuss migration plan for existing pipelines (if applicable)
  - Identify rollback strategy
- **Output**: Team consensus to adopt

#### Stage 7: Adopt

- **Input**: Team consensus from Stage 6
- **Action**:
  - Update CI/CD pipelines if needed
  - Deploy to production via normal release process
  - Monitor closely for 1 week post-deployment
  - Create retrospective if issues arise
- **Output**: Feature in production, tracked in change log

## 5. Patch and Breaking Change Tracking

### 5.1 Patch Tracking

Patches (minor version updates, bug fixes) are tracked but don't require the full feature evaluation process.

| Item | Tracking Method | Review Frequency |
|---|---|---|
| Databricks Runtime patches | DBR release notes page | Monthly + ad-hoc |
| Delta Lake patches | GitHub releases page | Monthly |
| Azure SDK patches | NuGet / pip release feed | Weekly (automated by Dependabot) |
| ADF service updates | Azure updates page | Monthly |
| Python library patches | pip audit / Dependabot | Weekly (automated) |

### 5.2 Breaking Change Assessment

When a breaking change is identified:

1. **Assess Impact**: Which pipelines, notebooks, or services are affected?
2. **Document**: Create a migration document with before/after examples
3. **Plan**: Schedule migration work as a feature branch with dedicated testing
4. **Communicate**: Notify all team members and stakeholders
5. **Execute**: Deploy during a maintenance window with rollback plan

### 5.3 Breaking Change Log Template


## 8. Summary

This document is the single source of truth for official documentation links. It must be:

1. **Reviewed monthly** — First Monday of each month, 2 hours, rotating reviewer
2. **Link-validated** — Run `link_checker.py` to verify all URLs are live
3. **Change-logged** — Every review produces a dated entry in the Change Log
4. **Feature-aware** — New features follow the 7-stage evaluation process (Monitor → Assess → Spike → System Test → Document → Review → Adopt)
5. **Breaking-change-aware** — Breaking changes get a migration document, timeline, and rollback plan

Stale documentation is worse than no documentation. Treat this document as a living artifact that is actively maintained, not a one-time deliverable.
